# Agent State · Structure

## 1. 文档目的

本文件用于定义 **Agent State 的结构表达方式**。

它回答的问题只有一个：

> **Agent 如何以一种稳定、可扩展的结构，
> 来表达“此刻自身所处的整体运行状态”？**

本文件不涉及：

- State 的生成或更新机制
- State 的读取策略或使用规则
- 任何流程、阶段或生命周期设计

---

## 2. State 的本质形式

Agent State 是一个 **结构化的整体状态快照（snapshot）**。

它描述的是在某一时刻：

- Agent 所承载的系统性运行条件
- Agent 所处的全局约束环境

而不是：

- 当前正在执行的步骤
- 某个模块的内部状态
- 某次任务的处理进度

State 的存在 **不依赖于任务是否存在**。

---

## 3. 结构总览（抽象形态）

在结构层面，Agent State 可以被抽象表示为：

```
AgentState {
    dimension_1,
    dimension_2,
    dimension_3,
    ...
}
```

其中每一个 `dimension`：

- 描述一个**相对独立的运行条件维度**
- 不表达意图或决策
- 不具备控制语义
- 可与其他维度同时成立

Agent State **不是单一值**，
而是由多个状态维度共同构成的组合体。

---

## 4. 示例结构（示意性）

以下结构仅用于说明 **Agent State 应当“长什么样”**，
而非规定具体字段或实现细节。

```
AgentState {
    cognitive_load:        Float   // 当前认知负载强度
    attention_level:       Float   // 可用注意力水平
    interruption_tolerance: Float  // 可被打断程度
    execution_conservatism: Float  // 对外部副作用的保守程度
    system_pressure:       Float   // 系统整体压力或拥塞水平
}
```

这些字段具备以下共同特征：

- 值域是连续的（如区间或强度）
- 语义描述“条件是否成立到什么程度”
- 不直接对应任何动作或决策结果

---

## 5. 状态维度的设计原则

### 5.1 维度之间相互独立

任意一个 State 维度：

- 不应依赖其他维度才能成立
- 不应隐含“阶段顺序”或“前后关系”

多个维度可以自然叠加，
共同构成当前的整体运行背景。

---

### 5.2 维度表达“状态”，而非“模式”

State 维度不应表达：

- “正在执行”
- “准备中”
- “已完成”
- “等待中”

这些都属于流程或生命周期语义。

State 维度只描述：

> **Agent 此刻处在什么样的运行条件之下。**

---

### 5.3 维度是描述性的，而非命令性的

State 中的任何字段：

- 不应被解读为“必须做什么”
- 不应直接触发行为
- 不应被当作控制信号

它们只提供背景事实。

---

## 6. 时间与演化特性

Agent State 是时间相关的，但不是流程驱动的。

- State 会随运行自然变化
- 变化可以是渐进的、连续的
- 不要求显式切换或跳转

State 的变化不等价于：

- 流程推进
- 阶段迁移
- 生命周期事件

---

## 7. 与其他概念的严格边界

Agent State **不等价于**：

- Runtime 的调度状态
- Task / Action 的生命周期
- Cognition 的推理阶段
- 控制流或状态机节点

如果某个字段开始表达“现在该做哪一步”，
它就已经越界。

---

## 8. 设计底线（强约束）

- Agent State **不得被建模为有限状态机**
- Agent State **不得以枚举主状态存在**
- Agent State **不得携带控制或决策语义**
- Agent State **不得隐式描述流程阶段**

这些约束是为了确保：

> State 始终是“像人体状态一样的背景条件”，
> 而不是一个隐形的控制系统。

---

## 9. 小结

Agent State 的结构设计目标是：

> **以多维、连续、可组合的方式，
> 表达 Agent 此刻所处的整体运行状态。**

它始终存在，
始终被系统各处参考，
却不直接推动任何一个具体行为。

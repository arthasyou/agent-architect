# Runtime · Input Event Model

## 1. 输入模型的设计前提

Runtime 是时间与因果裁决系统，
而非语义理解或任务调度系统。

因此，Runtime 接收的不是“模块输出”，
而是**已经发生、且可能影响时间线与因果合法性的事件**。

所有进入 Runtime 的输入，
都必须满足以下基本前提：

- 事件已经发生，不可回滚
- 事件具有明确的时间归属
- 事件可能影响当前因果时间段的成立性

不满足以上前提的内容，不应进入 Runtime。

---

## 2. Runtime 接收的输入事件类型

Runtime 只接收以下三类输入事件。

### 2.1 感知事件（Perception Events）

感知事件表示：
**外部世界的某个变化，已被感知系统确认并编码完成。**

Runtime 对感知事件的最小假设是：

- 事件已发生
- 事件属于某一因果时间段
- 事件可能否定当前世界前提

Runtime 不理解感知事件的语义内容，
只关心其是否对当前因果假设构成挑战。

---

### 2.2 认知结果事件（Cognition Result Events）

认知结果事件表示：
**一次完整认知过程已经结束，并产生了一个可供裁决的结果。**

Runtime 对认知结果事件的最小假设是：

- 认知过程已完成，不可中断
- 结果可能具有现实影响
- 结果不保证会被采用

Runtime 不接收认知过程本身，
只接收已经完成的认知结果。

---

### 2.3 行为状态事件（Action State Events）

行为状态事件表示：
**某个已存在的行为实例，其执行状态发生了变化。**

包括但不限于：

- 行为启动
- 行为暂停 / 恢复
- 行为完成
- 行为失败
- 行为前提被外部事件否定

Runtime 对行为状态事件的最小假设是：

- 行为跨时间存在
- 行为具有外部副作用
- 行为必须接受持续的合法性裁决

---

## 3. Runtime 明确不接收的输入

为防止职责膨胀，Runtime 明确不接收以下内容：

- 原始或未编码的感知数据
- 认知过程中的中间状态
- 认知推理的中间结论
- 语义解释或重要性判断
- 记忆系统的读请求
- 策略评估、优先级计算结果

上述内容若进入 Runtime，
将使 Runtime 越权进入语义或策略层。

---

## 4. Memory 与 Control State 的输入地位说明

Memory 与 Control State **不作为 Runtime 的输入事件**。

它们在 Runtime 中的角色是：

- **Memory**：作为历史背景，在裁决阶段被查询
- **Control State**：作为裁决约束条件，影响裁决结果

二者均不触发因果时间段的推进，
也不构成 Runtime 的驱动事件。

---

## 5. 输入模型的边界总结

Runtime 的输入模型遵循以下边界原则：

- 输入必须是“事件”，而非“状态查询”
- 输入必须可能影响因果裁决
- 输入不携带语义解释责任

Runtime 接收事件，
但不理解事件。

---

本节仅定义 Runtime 接收**哪些输入**，
不定义 Runtime **如何在不同时间状态下处理这些输入**。
